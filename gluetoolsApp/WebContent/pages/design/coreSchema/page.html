The data within a GLUE project is structured according to the Core Schema; a fixed set of object types and relationships. These bring a certain level of evolution-oriented organization to virus nucleotide sequence data and also capture the objects and relationships most commonly required to utilise the data.

In the text below, we outline the main GLUE object types, their intended semantics and the relationships between them. GLUE object types are denoted in italicised CamelCase, e.g. FeatureLocation.

Sequences and Sources

A set of FASTA virus nucleotide sequences forms the foundation of each GLUE project data set. Each GLUE Sequence object is assumed to be a section of viral RNA or DNA. Sequences may originate from any laboratory sequencing technique as long as FASTA nucleotide data is produced. Sequences may also be in silico constructs, such ancestral reconstructions, or consensus sequences which are each representatives of a larger set. A Sequence object may cover the whole of the virus genome, or only a part, and may be have unknown subregions, indicated by the ‘N’ character. For endogenous viruses a Sequence may cover some local areas of host nucleotide sequence, beyond the virus genome.

Within a GLUE project, Sequences must be stored within Source objects. These exist purely to group sequences; the meaning or usage of these groups is left to the project developer. Sequences are identified by their sequenceID string field. However, the uniqueness of the sequenceID value is limited to the containing Source object. So, no two Sequences within the same Source may have the same sequenceID; conversely two Sequences within two different Sources may have the same sequenceID.

Sequences may be imported into GLUE from multiple file formats. The specific format is recorded in a field value on the Sequence object, and the original file data is stored in the database in byte form. The aim of this is to allow retrieval of any non-nucleotide data which may be present in the original file. The original file, including the nucleotide data is immutable for the lifetime of the Sequence object.

Features

For many virus species, biological properties of certain named regions of the virus genome have been suggested or established. Within a GLUE project, Feature objects may be created for any genomic region of interest. Examples include regions which are understood to code for polyproteins or individual proteins, or regions which code for domains within a protein. Feature objects may also represent non-coding regions such as promoter regions. The Feature objects in a GLUE project may be arranged in a hierarchy, with each Feature optionally having a single parent Feature object and several child Feature objects associated. This should reflect the containment relationships of the corresponding genome regions.

Feature objects may also have FeatureMetatag objects associated. These are simple metadata properties, with fixed semantics, which allow specialized GLUE functionality to interpret Feature objects, for example to perform protein translation.

ReferenceSequences and FeatureLocations

GLUE uses ReferenceSequence objects to organize, link and interpret the wider set of Sequence objects. A ReferenceSequence is based on a specific Sequence and this relationship is mandatory and immutable for the lifetime of the ReferenceSequence. The choice of which Sequence objects to use for ReferenceSequence objects will vary from project to project, some examples are:

  • Sequences which have been curated as reference sequences by another database, e.g. NCBI RefSeq
  • Sequences derived from well-known virus strains which were identified early in the study of a particular virus species or subtype
  • Sequences derived from virus strains which have become established as having important laboratory uses
  • Sequences where the genome is particularly well-annotated. Less frequently, sequences derived from an in silico procedure such as computing the consensus of a wider set, or ancestral reconstruction

A ReferenceSequence object may contain FeatureLocation objects in order to represent the presence and position of a specific Feature on the ReferenceSequence. A FeatureLocation is considered a wholly contained constituent part of a ReferenceSequence, but it is also immutably associated with a single Feature during its lifetime.

A certain Feature may not be present (i.e. be associated with a FeatureLocation) on every ReferenceSequence in the GLUE project. This flexibility may be useful for example when specific virus genes exist only in some taxa within the project scope.

A FeatureLocation contains a set of FeatureSegment objects which specify the coordinates of the relevant Feature on the ReferenceSequence. The FeatureSegment objects contain start and end points, and together may specify a discontiguous region.

Alignments and AlignmentMembers

One key aspect which GLUE projects aim to capture is the representation of nucleotide homologies between Sequence objects. At a fine-grained level, these homologies may be thought of as statements proposing that a certain segment of nucleotides in one Sequence has the same evolutionary origin as a certain segment in a second Sequence.

Alignment objects in GLUE aggregate homologies relating to a set of “member” Sequence objects that are associated to the Alignment via AlignmentMember objects. So in concrete terms an Alignment object contains a set of AlignmentMember objects and each AlignmentMember associates a member Sequence with the containing Alignment.

Within an Alignment all homologies use the same reference coordinate space. Each AlignmentMember in turn contains a set of AlignedSegment objects, which are each statements of homology between the member Sequence nucleotides and the reference coordinate space of the Alignment. To represent this, the AlignedSegment is a tuple with four integer fields:

<refStart, refEnd, memberStart, memberEnd>

This states that the region [memberStart, memberEnd] in the member Sequence is homologous with the region [refStart, refEnd] in the reference coordinate space of the containing Alignment. An invariant maintained by GLUE is that these two regions must be of equal length.

This reference coordinate space associated with an Alignment object is used as an indirect step relating the member sequence nucleotides with each other. So, two blocks of nucleotides from distinct member Sequence objects within an Alignment are considered to be homologous when they are mapped via AlignedSegment objects to the same reference coordinates.

Alignment objects may be unconstrained or constrained. Unconstrained Alignments are used in GLUE projects to store the results of any multiple sequence alignment process aimed at identifying homologies between sequences. This includes classical algorithms such as MUSCLE or MAFFT but also manual alignment techniques. In unconstrained alignments the reference coordinate space is purely notional, and not based on any particular Sequence. Nucleotide position columns in an this coordinate space may be added in an unrestricted way in order to accommodate any pairwise homology between member Sequence objects which the alignment process proposes.

Sequence objects may be members of multiple Alignments, via different AlignmentMember associations. One application of this is where separate unconstrained Alignment objects are used to represent homologies in different regions of the virus genome. This may be appropriate for example if the best alignment technique differs from one genomic region to another.

The main function of a ReferenceSequence in GLUE is to provide a concrete coordinate space in which the nucleotide data in Sequence objects may be interpreted. As an alternative to being unconstrained, Alignment objects may be constrained to a ReferenceSequence. In this case, the nucleotide Sequence underlying the constraining ReferenceSequence provides the reference coordinate space for the constrained Alignment. Therefore, AlignedSegment objects within a constrained Alignment propose homology between a nucleotide block on the member Sequence and an equal-length block on the constraining ReferenceSequence.

The AlignedSegment objects within constrained Alignment objects may be derived from those within unconstrained Alignment objects where both the member and reference sequences are present. However, where the member sequences of the constrained Alignment are known to be closely related to the ReferenceSequence, another possibility exists. In this case the constrained Alignment homologies may be computed using a simple pairwise technique between the member and reference sequence, for example based on BLAST.
